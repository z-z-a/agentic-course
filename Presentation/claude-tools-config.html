What is the .claude/ Folder?

  • A config directory in your project root — or ~/.claude/ for global settings
  • Claude reads it automatically at the start of every session
  • Commit it to git — your whole team benefits

CLAUDE.md — The Instruction File

  • Always loaded into Claude's context at session start
  • Project CLAUDE.md → team conventions, tech stack, rules
  • Global ~/.claude/CLAUDE.md → your personal defaults
  • CLAUDE.local.md → personal project overrides (gitignored)

Example content

# My Project
## Stack
- Next.js, TypeScript, Postgres

## Rules
- Always write tests for new functions
- PR descriptions must include a "Test plan" section
- Never use `console.log` — use the logger utility
Think of it as your team's onboarding doc — written once, read by Claude every time.

settings.json — Permissions & Config

  • Controls what Claude can do without asking
  • Committed to git → shared with the whole team
  • settings.local.json for personal overrides (gitignored)
{
  "model": "claude-sonnet-4-6",
  "permissions": {
    "allow": ["Bash(git *)", "Bash(npm test)", "Read"],
    "deny":  ["Bash(rm -rf *)"]
  },
  "env": {
    "NODE_ENV": "development"
  },
  "hooks": {
    "PostToolUse": ["echo 'tool finished'"]
  }
}

Custom Commands — The Basics

How it works

  • Create a .md file in .claude/commands/
  • The filename → the slash command name
  • The file content → the prompt Claude runs
review.md   →  /review
standup.md  →  /standup
deploy.md   →  /deploy

Two scopes

ScopePath
Project .claude/commands/
Personal ~/.claude/commands/

Personal commands are available in every project on your machine.

Writing Your First Custom Command

.claude/commands/standup.md

Look at my git log for the last 24 hours
and write a standup update.

Format:
- What I did yesterday
- What I'm doing today
- Any blockers

Keep it under 5 bullet points total.

Usage: type /standup in any session

.claude/commands/explain-pr.md

Read the current git diff against main
and write a pull request description.

Sections to include:
1. Summary
2. Changes Made
3. Test Plan

Be concise and use bullet points.

Usage: type /explain-pr in any session

Organizing Commands in Subdirectories

Commands can be nested — the path becomes part of the display name in the / menu:

.claude/commands/
├── frontend/
│   ├── component.md   →  /component   (tagged "project:frontend")
│   └── storybook.md   →  /storybook   (tagged "project:frontend")
├── backend/
│   ├── api-test.md    →  /api-test    (tagged "project:backend")
│   └── migration.md   →  /migration   (tagged "project:backend")
└── deploy-check.md    →  /deploy-check
Useful for large monorepos — keeps commands discoverable without polluting a flat list.

Summary & Best Practices

Key takeaways

  1. .claude/ is your project's Claude config — commit it to git
  2. CLAUDE.md = team instructions; CLAUDE.local.md = personal overrides
  3. One .md file = one slash command — keep prompts focused
  4. Start with commands/, move to skills/ when you need auto-invocation
  5. ~/.claude/commands/ for personal commands across all projects

Getting started checklist

  • Run /init to auto-generate CLAUDE.md
  • Add team conventions to CLAUDE.md
  • Create 2–3 commands for your most repeated prompts
  • Pre-approve safe Bash commands in settings.json
  • Share the .claude/ folder with your team via git
