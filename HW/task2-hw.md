# תרגיל 2: מימוש שיטתי עם לולאת PIV

## סקירה כללית

זהו התרגיל השני בקורס לקידוד אגנטי. לאחר שחוויתם את זרימת העבודה הטבעית שלכם בתרגיל 1, כעת תיישמו את **לולאת PIV** - גישה שיטתית לקידוד בעזרת AI שמשפרת באופן דרמטי את איכות הפלט ונותנת לכם יותר שליטה על המימוש.

תממשו את אותה תכונת סינון מוצרים מתרגיל 1, אך הפעם תשתמשו במה שלמדתם במודול 2: מתודולוגיית לולאת PIV.

**לולאת PIV**: **P**lanning (תכנון) → **I**mplementing (מימוש) → **V**alidating (אימות) → חזרה על התהליך

> שימו לב שהמימוש הזה פשוט מספיק כך שמדובר באמת רק באיטרציה אחת של ה"לולאה".

## התחלה מהירה

אם עדיין לא הגדרתם את האפליקציה מתרגיל 1, הריצו אותה בשני טרמינלים:

**טרמינל 1 - Backend**:
```bash
cd app/backend
uv venv --python 3.12
uv sync
uv run python run_api.py
# → http://localhost:8000
```

**טרמינל 2 - Frontend**:
```bash
cd app/frontend
npm install -g bun
bun install
bun dev
# → http://localhost:3000
```

## מטרות

1. **ליישם את מתודולוגיית לולאת PIV** על מימוש fullstack אמיתי
2. **לחוות את ההבדל** בחלקות התהליך, השליטה והביטחון לעומת תרגיל 1
3. **ללמוד ליצור הקשר עשיר** שמכוון את ה-AI למימושים טובים יותר
4. **להישאר בכיסא הנהג** תוך האצלת עבודת הקידוד לעוזר ה-AI שלכם

**על מה לשים לב**: המימוש עצמו אמור לצאת בערך אותו דבר כמו בתרגיל 1 - כבר דיברנו על פשטות המשימה. במקום זאת, התמקדו ב:
- כמה חלק יותר לולאת PIV הופכת את התהליך?
- כמה יותר שליטה יש לכם על פרטי המימוש?
- כמה טוב יותר אתם מבינים מה נבנה?
- האם אתם מכוונים את ה-AI, או שה-AI מכוון אתכם?

## המשימה

> פתחו את עוזר הקידוד ה-AI שלכם בתיקיית `app/` בשביל זה!

תממשו את אותה תכונת סינון מוצרים מתרגיל 1, אך הפעם באמצעות מתודולוגיית לולאת PIV.

### חלק 1: צרו את תוכנית המימוש שלכם

התחילו עם מפרטי המשימות המובנים שסופקו:

- **משימת Backend**: `tickets/KOD-1.md`
- **משימת Frontend**: `tickets/KOD-2.md`

משימות אלו כתובות בפורמט שאתם עשויים לקבל מכלי ניהול פרויקטים כמו Linear, Jira או Asana. הן כוללות דרישות, קריטריוני קבלה, הערות טכניות והגדרת "סיום". הכללת זה בהקשר/תוכנית שלכם זה יישום חלק ה-RAG של הנדסת הקשר!

השתמשו במפרטי המשימות האלה כדי ליצור תוכנית מימוש מובנית שיכולה לכלול חלקים כמו:

1. **קריטריוני הצלחה**: איך תדעו שסיימתם?
   - כל הטסטים עוברים
   - ממשק ה-Frontend מספק את כל פקדי הסינון הנדרשים
   - פריטי רשימת הבדיקה הידנית עוברים

2. **תיעוד לעיון**:
   - דפוסי קוד קיימים ב-`app/backend/app/`
   - דפוסי frontend קיימים ב-`app/frontend/src/`
   - קבצי טסטים שמראים התנהגות צפויה

3. **פירוק המשימה**:
   - Backend: מודל פרמטרים, פונקציית שירות, עדכוני endpoint
   - Frontend: רכיב סינון, עדכוני לקוח API, ניהול state

4. **מבנה קוד רצוי**:
   - אילו מוסכמות שמות להשתמש
   - איזה ארגון קבצים להשתמש
   - אילו דפוסי logging לממש
   - איזו type safety לשמור

### חלק 2: מימוש באמצעות התוכנית שלכם

עם התוכנית שיצרתם, בקשו מעוזר הקידוד ה-AI לממש את התכונות:

- ספקו את התוכנית המלאה שלכם כהקשר בעת הפנייה לעוזר ה-AI
- הפנו לחלקים ספציפיים בתוכנית שלכם בכל שלב
- שמרו על ה-AI ממוקד בגישה הרצויה שלכם
- שימו לב כמה יותר שליטה יש לכם בהשוואה לתרגיל 1

### חלק 3: אימות וחזרה על התהליך

אתם בעצמכם:

- חזרו על כל בעיה שנמצאה

## הערות חשובות

### השוואת תהליכים

המטרה היא לא רק להשלים את התכונה - אלא **לחוות את ההבדל** בין זרימת העבודה שלכם בתרגיל 1 לגישת לולאת PIV. היו כנים לגבי:

- שליטה על המימוש (האם הרגשתם יותר בשליטה?)
- הבנת הקוד (האם אתם מבינים אותו טוב יותר?)
- ביטחון באיכות (האם אתם יותר בטוחים?)

### אל תדלגו על שלב התכנון

זה עלול להרגיש כמו עבודה נוספת מראש, אבל מכאן מגיע הכוח של לולאת PIV:

- **התאמת AI טובה יותר**: ל-AI יש הקשר ברור לגבי מה שאתם רוצים
- **פחות הפתעות**: חשבתם על הגישה לפני שהקוד מופיע
- **דיבאגינג קל יותר**: אתם יודעים מה אמור לקרות, אז טעויות בקוד נהיות יותר ברורות
- **יותר שליטה**: אתם מכוונים את ה-AI עם התוכנית שלכם, לא מגיבים להצעות שלו
